# claude-beginner-harness-plugin

## 플러그인 목표

Claude Code를 처음 사용하거나 익숙하지 않은 사람들이 **좋은 습관을 만들어가도록 돕는 하네스 플러그인**입니다.

토큰 사용량이나 작업 시간이 다소 늘어나더라도, 처음부터 올바른 방식으로 Claude와 협업하는 패턴을 익히는 것을 우선합니다.

## 핵심 철학

- **속도보다 습관**: 빠르게 결과를 얻는 것보다, 올바른 과정을 반복해서 몸에 익히는 것이 중요합니다.
- **추측하지 않기**: 불명확한 요구사항으로 작업을 시작하면 방향이 틀린 결과가 나옵니다. 먼저 충분히 이해하고 작업을 시작합니다.
- **초보자 친화적**: Claude Code의 기능을 잘 모르는 사람도 좋은 결과를 낼 수 있도록 안내합니다.

## 포함된 스킬

### `init` — CLAUDE.md 초기화

프로젝트 루트에 CLAUDE.md가 없을 때, 어떤 작업보다 먼저 CLAUDE.md 생성을 최우선으로 처리합니다. CLAUDE.md에는 프로젝트 개요만 간략히 담고, 세부 규칙은 `.claude` 폴더 문서를 참조하도록 작성합니다.

- **자동 활성화**: 코딩 작업 요청 시 프로젝트 루트에 CLAUDE.md가 없는 경우
- **수동 호출**: `/init`

### `scope` — 바이브코딩 범위 판단

코딩 작업 시작 전에 AI가 자율적으로 진행해도 되는 작업인지, 사용자와 협업 설계가 필요한 작업인지 판단합니다. `clarify` 이전에 항상 먼저 실행됩니다.

- **바이브코딩 영역**: 일회성 마케팅 페이지, 실험/프로토타입, 데모, 중요도 낮은 독립 코드 → 질문 없이 자율 진행
- **협업 설계 영역**: 프로덕션 코드, 보안 관련, 성능 최적화, 핵심 추상화, 데이터 모델 변경 → clarify/preplan 프로세스 진행
- **자동 활성화**: 모든 코딩 작업 요청 시 Claude가 스스로 판단
- **수동 호출**: `/scope`

### `clarify` — 요구사항 구체화

작업 요청이 모호하거나 AI가 추론을 해야 하는 상황에서 즉시 코드 작업을 시작하는 대신, 질문지를 통해 요구사항을 먼저 구체화합니다. **협업 설계 영역으로 판단된 작업에만 적용**됩니다.

- **자동 활성화**: scope 판단 결과 협업 설계 영역이고, 목적·범위·입출력이 불명확한 경우
- **수동 호출**: `/clarify`

### `preplan` — 작업 시작 전 준비

컨텍스트가 없는 상태에서 복잡한 작업을 시작하기 전, 도메인 용어를 정립하고 필요 시 계획 모드 진입을 권유합니다.

- **자동 활성화**: 초기 컨텍스트 없이 복잡한 작업 요청 시 Claude가 스스로 사용
- **계획 모드 권유**: 여러 파일/모듈에 걸친 변경, 기존 모듈 연동 신규 기능 개발, 리팩토링
- **수동 호출**: `/preplan`

### `input` — INPUT.md 읽기

`/input` 호출 시 프로젝트 루트의 `INPUT.md`를 읽고 그 내용에 대해 응답합니다. INPUT.md는 버전 관리 시스템 ignore 파일에 자동 등록됩니다.

- **수동 호출**: `/input`

### `capture` — 대화 내용 저장

마지막 사용자 입력과 Claude의 응답을 `.claude/history/`에 타임스탬프 파일명으로 저장합니다.

- **수동 호출**: `/capture`

### `worktree` — Git Worktree 격리 개발

기능 규모가 크거나 이전 작업과 완전히 다른 작업을 시작할 때 git worktree 생성을 권고합니다. 관련 MCP 도구(`EnterWorktree`)가 있으면 즉시 worktree를 만들어 격리된 환경에서 안전하게 개발할 수 있도록 합니다.

- **자동 활성화**: 변경 파일이 많은 경우, 핵심 공통 모듈 수정, 이전 작업과 무관한 새 기능 시작 시
- **MCP 연동**: `EnterWorktree` 도구가 있으면 사용자 승인 후 즉시 worktree 생성
- **수동 호출**: `/worktree`

### `impact-check` — 변경 영향 분석 및 계약 갱신

기존 코드나 요구사항을 변경하기 전에, 영향받는 영역을 파악하고 암묵적 전제를 명시적으로 갱신합니다. 경고가 아니라 **변경 전/후의 전제를 함께 확인하고 합의**하는 방식으로 진행합니다.

- **자동 활성화**: 기존 코드/요구사항에 대한 변경 요청 시 (인터페이스, 동작, 데이터 모델, 로직 변경)
- **clarify 연동**: `/clarify` 결과물이 있으면 해당 합의를 계약 기준으로 활용
- **수동 호출**: `/impact-check`

### `handoff` — 컨텍스트 관리 및 인계

대화가 길어져 컨텍스트가 50% 이상 채워졌을 때, 작업 흐름을 끊지 않고 컨텍스트를 초기화할 수 있도록 HANDOFF.md 인계 문서를 작성합니다.

- **자동 활성화**: 컨텍스트 사용량이 50% 이상으로 추정될 때 Claude가 스스로 사용
- **새 작업 감지**: HANDOFF.md가 1시간 이상 경과하거나 완전히 다른 작업 시작 시 초기화 권유
- **gitignore 처리**: 버전 관리 시스템이 있으면 HANDOFF.md 생성 시 자동으로 ignore 파일에 등록
- **기획 문서 분리**: 장기 참조가 필요한 기획/설계 내용은 `.claude/docs/`에 별도 문서로 작성, 코드 세부 사항은 코드와 주석으로 직접 파악
- **수동 호출**: `/handoff`

## 개발 원칙

이 플러그인에 새로운 스킬이나 기능을 추가할 때는 아래 기준을 따릅니다:

1. 초보자가 자주 겪는 실수나 비효율적인 패턴을 해결하는가?
2. 단순히 편의를 제공하는 것이 아니라, 좋은 습관 형성에 기여하는가?
3. 작업 결과보다 작업 과정을 개선하는가?
